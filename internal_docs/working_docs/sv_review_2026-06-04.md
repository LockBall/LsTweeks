### Known Issues (from prior sv_review)

**SL-0a** `sl_core.lua:26` — `_PlaySound(soundkit_id, channel)` channel-type mismatch.
`C_Sound.PlaySound` expects `uiSoundSubType?` (an enum, e.g. `Enum.SoundType.SFX`), not a raw string like `"SFX"`. The global `PlaySound` fallback also accepts an optional channel string, but the string form `"SFX"` may not match the subtype enum used by the API. This is a potential silent no-op in the SFX channel — sound falls back to default routing instead of playing on SFX. Review: confirm what `SOUNDKIT`-based playback uses in-game and whether the channel arg matters here. Low risk for in-game behavior since `SOUNDKIT` paths rarely rely on the explicit channel, but worth verifying against Ketho's `SoundDocumentation.lua`.

**SL-0b** `sl_fishing.lua:46` — `raw_value = get_cvar(channel.cvar)` LuaLS cast-local-type warning.
`get_cvar` is initialized as `local raw_value = M._fishing_focus_active and ...` where the left side of `and` is boolean-typed at LuaLS analysis time. LuaLS narrows `raw_value` to `boolean | string | nil` but the later `tonumber(raw_value)` call is safe at runtime. No behavioral bug; cosmetic/annotation issue only.

### New Issues

#### Dead Code

**SL-1** `sl_functions.lua:68-75` — `get_replacement_paths_for_preset` always returns a single-element table or nil.
The function wraps one path in `{ path }` and the caller `get_next_replacement_path` always reads `paths[1]`. The intermediate table allocation serves no purpose: no multi-path logic exists anywhere in the module, and `rebuild_event_cache` builds its own slots directly (bypassing this function). The table wrapper is dead structure — `get_replacement_paths_for_preset` could return the string directly, or `get_next_replacement_path` could inline the lookup. Not a CPU hotpath concern (only called on preview/apply, not per-frame), but it is misleading and wasteful.

**SL-2** `sl_functions.lua:93-100` — `get_current_sound_channel_percent` is a one-line passthrough.
```lua
function M.get_current_sound_channel_percent(channel)
    return read_channel_percent(channel)
end
```
`read_channel_percent` is a module-local already. This public wrapper adds zero behavior, is called from one site in `sl_gui.lua:430`, and is not referenced in `sl_fishing.lua`. Either inline it at the call site or remove the wrapper.

**SL-3** `sl_core.lua:31-58` — `play_original_file` has a dead multi-file cycling path.
Lines 43-46 handle `file_count > 1` by cycling through `original_file_ids`. No current target has more than one `original_file_id` (Achievement: 1, Ready Check: 1), and `apply_replacement_paths` / `build_numbered_replacement_paths` are also only per-target. The cycling state `target._original_cycle_index` is written to the static metadata table. The cycle branch is unreachable with current data and adds noise. Consider removing unless multi-ID targets are planned.

**SL-4** `sl_defaults.lua:23-32` — `SOUND_ASSETS` nil-guard inside `build_numbered_replacement_paths` is always true.
```lua
local asset = M.SOUND_ASSETS and M.SOUND_ASSETS[asset_key]
```
`M.SOUND_ASSETS` is defined on line 12, two lines before this function. The `M.SOUND_ASSETS and` guard is always true at call time; `apply_replacement_paths` is only called after both are assigned. Minor dead condition, not a bug.

#### Repeated Code / Consolidation Opportunities

**SL-5** `sl_fishing.lua` — `get_cvar` / `set_cvar` are private locals duplicated in spirit with any other future file that needs CVar access.
These are clean, properly abstracted, and correctly handle the `C_CVar` → `GetCVar` fallback. No issue in isolation. Worth noting if a third file ever needs CVar access — at that point they should be promoted to `sl_functions.lua` or a shared util. Track for future.

**SL-6** `sl_gui.lua:384-408` — custom tooltip implementation (inline backdrop + fontstring + show/hide).
This is a one-off tooltip built by hand for the "Enable Fishing Focus" checkbox. It replicates the same backdrop constants used in `apply_box_backdrop` (slightly different colors). If the pattern recurs, consolidate into a small helper. One instance is fine as-is; document it so it doesn't get copy-pasted.

**SL-7** `sl_gui.lua:46-55` — `apply_box_backdrop` is a local helper defined once, used twice (line 121: `target_list_panel`, line 122: `slider_panel`).
This is already consolidated correctly. No action needed.

**SL-8** `sl_main.lua:25-48` — control resync loop in `on_reset_complete` is manual per-key iteration.
Three separate `M.controls[target_key .. "_*"]` lookups per target. With only two targets this is fine. If targets grow, consider a small helper that syncs one target's controls from its DB. Not a current problem.

#### CPU / Performance Issues

**SL-9** `sl_functions.lua:10-22` — `get_db()` calls `apply_defaults` once per session (guarded by `M._defaults_applied`) — correct and intentional. No issue.

**SL-10** `sl_functions.lua:24-42` — `get_target_db()` calls `addon.apply_defaults` **every call** with no guard.
Unlike `get_db()`, `get_target_db` has no `_defaults_applied`-style guard. It is called from `rebuild_event_cache` (once per `apply_sound_levels`), from `on_reset_complete`, and from the slider's `OnValueChanged` (via `apply_sound_levels`). At current call volume this is not a CPU hotpath, but it is inconsistent with the module's stated pattern and could degrade if called more frequently. The preset validation and `use_original` correction at lines 35-41 make this non-trivial to guard without separate state. Low priority but worth noting.

**SL-11** `sl_functions.lua:83-99` — `get_ordered_sound_targets()` allocates a new table and sorts on every call.
Called from `build_sounds_tab` once at GUI build time. Not a performance issue at all — just noting it is not cached. If called from a hot path in future this would need a cached result. Fine as-is.

**SL-12** `sl_core.lua:133-151` — `handle_event` hot path is clean.
One table lookup, array walk over a small slot list, direct `_PlaySoundFile` / `_PlaySound` call, no DB access, no `apply_defaults`. This is the performance-critical path and it is well-designed. No issues.

**SL-13** `sl_fishing.lua:169-176` — `handle_fishing_focus_event` guards `unit ~= "player" or spell_id ~= FISHING_CHANNEL_SPELL_ID` correctly on registered unit events.
`RegisterUnitEvent("...", "player")` already filters by unit, so the `unit ~= "player"` guard is technically redundant. Not a bug; minor micro-optimization opportunity. The `spell_id` check remains essential.

**SL-14** `sl_fishing.lua:11` — `_PlaySound` and `_StopSound` are re-resolved in `sl_fishing.lua` independently of `sl_core.lua`.
Both files resolve their own upvalue locals at load time. The `sl_fishing.lua` copies (`_PlaySound`, `_StopSound`) are only used for Fishing Focus preview. This duplication is intentional and correct — each file is self-contained for its audio calls. No issue; matches the stated pattern in `proj_mem`.

#### Minor / Style

**SL-15** `sl_defaults.lua:37-43` — `apply_replacement_paths` iterates `targets or {}` with `pairs` — correct since `SOUND_TARGETS` is a keyed table, not an array. Fine.

**SL-16** `sl_gui.lua:591` — Tab positioning uses a ternary-style inline expression for `SetPoint`:
```lua
tab:SetPoint(i == 1 and "TOPLEFT" or "LEFT", i == 1 and parent or tabs[i-1], i == 1 and "TOPLEFT" or "RIGHT", i == 1 and UI.pad_x or 5, i == 1 and UI.pad_y or 0)
```
Five inline ternaries on one line. Readable but dense. No bug; style only.

**SL-17** `sl_gui.lua:346-348` — `focus_defaults` table is declared and populated via `refresh_current_values()` but is only used as the `reset_values` arg to `CreateSliderWithBox` on line 463. The table is correctly shared by reference so `refresh_current_values` keeps it live. This pattern is subtle and works correctly — just unusual to have a "reset defaults" table that is dynamically recomputed from live CVars. Documenting it here so future reviewers don't mistake it for a bug.
