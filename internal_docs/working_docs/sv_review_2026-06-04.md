## Consolidated Findings

### 1. Runtime tick interval used a raw literal

**Severity:** Low / style  
**File:** `sv_main.lua`

Original issue:

```lua
local TICK_SECONDS = 0.2
local FILL_TEST_TICK_SECONDS = 0.05
```

`TICK_SECONDS = 0.2` matched `addon.UPDATE_INTERVALS.fifth_sec` but bypassed the shared timing buckets from `proj_mem.md`.

Resolution:
- Confirmed `core/init.lua` loads before `sv_main.lua`, so `addon.UPDATE_INTERVALS.fifth_sec` is available at file load.
- Replaced normal runtime `TICK_SECONDS = 0.2` with `addon.UPDATE_INTERVALS.fifth_sec`.
- Left `FILL_TEST_TICK_SECONDS = 0.05` module-local because no shared timing bucket exists and it drives smoother fill-test animation; added a short comment.

### 2. Stable layout refresh did unnecessary work each tick

**Severity:** Medium / CPU  
**File:** `sv_bar.lua`, with callers in `sv_main.lua`

Original issue:
- `M.refresh()` and `render_fill_test()` call `M.apply_layout()`.
- `apply_layout()` had a clean-layout guard, but only after `get_db()` and `M.ensure_frame()`.
- The layout-signature string was also built before the final signature comparison.

Resolution:
- Moved the clean-layout guard to the top of `M.apply_layout()`.
- Stable ticks now return before DB/frame lookup and before layout-signature construction.
- No separate code change was needed for `render_fill_test()` beyond this. Its repeated `M.set_move_mode(false)` and `set_frame_alpha(frame, 1)` calls already guard stable state.

### 3. ADDON_LOADED default application was redundant

**Severity:** Low / correctness  
**File:** `sv_main.lua`

Original issue:
- The `ADDON_LOADED` handler applied defaults, set `M._defaults_applied = true`, then called `get_db()`.
- `get_db()` also owns default application and normalization.
- This was safe, but the ownership was split.

Resolution:
- Simplified the loader so `get_db()` owns first default application and normalization.
- The handler still ensures top-level `Ls_Tweeks_DB` exists before calling `get_db()`.

### 5. `normalize_power_value()` parameter name implied clamping

**Severity:** Low / readability  
**File:** `sv_main.lua`

Original issue:
- `normalize_power_value(value, max_power, max_slots, power_type)` used `max_slots` as a scaling reference.

### 6. Slider callback read back from DB instead of receiving value

**Severity:** Medium / correctness  
**File:** `sv_gui.lua`, `functions/slider_with_box.lua`

Original issue:

```lua
local function set_setting_from_slider(key)
    return function()
        if M._syncing_slider_controls then return end
        local db = M.get_db and M.get_db()
        if not db then return end
        M.set_db_value(key, db[key])
    end
end
```

The review initially assumed `CreateSliderWithBox()` already passed the new value to callbacks. It did not; it called callbacks with no arguments.

Resolution:
- Updated `CreateSliderWithBox()` to pass the current numeric value through immediate reset callbacks and debounced slider callbacks.
- Existing zero-argument callbacks remain compatible in Lua.
- Updated Skyriding Vigor's `set_setting_from_slider()` to accept `value` and call `M.set_db_value(key, value)` directly.

### New findings from second pass

#### N1. `apply_layout()` still builds layout_signature string even when `_layout_dirty` is false but `_layout_signature` is nil (sv_bar.lua:477)

**Severity:** Low / edge case  
**File:** `sv_bar.lua:477`

The new early-exit guard is:
```lua
if not M._layout_dirty and M._layout_signature then
    return
end
```
This is correct: if either `_layout_dirty` is true OR `_layout_signature` is nil, it falls through to do layout work. The `_layout_signature = nil` path is the invalidation path (`invalidate_layout()` sets both dirty and nil), so falling through there is correct and intentional. No issue ÔÇö just noting the guard is deliberately a conjunction, not two separate checks.

#### N2. `slider_with_box.lua` reset button calls `run_callback(default_value)` directly when value is already at default (slider_with_box.lua:160)

**Severity:** Low / consistency  
**File:** `functions/slider_with_box.lua:157-161`

```lua
if slider:GetValue() ~= default_value then
    slider:SetValue(default_value)   -- triggers OnValueChanged Ôćĺ debounced_callback
else
    run_callback(default_value)      -- direct (immediate, no debounce)
end
```

When the slider is already at the default value, the reset button bypasses the debounce and calls `run_callback` immediately. When it is not at the default, `SetValue` fires `OnValueChanged` which goes through `debounced_callback`. The two paths are intentionally asymmetric (this is documented in proj_mem as a deliberate pattern: "Slider reset buttons must explicitly write the DB and run their callback even when the slider already displays the default value"). No bug, but the immediate path and the debounced path produce the same observable outcome here (both write DB through the slider binding and then call the callback). The behaviour is correct.

#### N3. `set_setting_from_slider` in sv_gui.lua now receives `value` from slider but x/y position sliders use `HookScript` with their own value parameter and do NOT go through `set_setting_from_slider` (sv_gui.lua:112-115, 129-132)

**Severity:** Low / consistency  
**File:** `sv_gui.lua:101-133`

The x and y position sliders do not use `set_setting_from_slider`. They use `HookScript("OnValueChanged", function(_, value) M.set_position_axis(..., value) end)` instead. This is intentional ÔÇö position axes use a different setter. However the pattern is now inconsistent: scale/spacing/fade_alpha use `CreateSliderWithBox` with a callback argument, while x/y use `CreateSliderWithBox` with no callback and a separate `HookScript`. Since `HookScript` fires after the internal `OnValueChanged` (which already writes `db_table[db_key]`), the position DB is written twice: once by the slider binding to `db.position`, then again by `M.set_position_axis`. For x/y this is redundant but harmless because `set_position_axis` re-reads `db.position[axis]` as its value when the argument is not nil. Not a bug; noted for awareness.

#### N4. `needs_progress_updates` fix: spell-charge fake duration still active in move mode (sv_main.lua:354-357, 373-374)

**Severity:** Low / logic nuance  
**File:** `sv_main.lua:354-374`

The fake charge injection in move mode sets `start_time = GetTime() - 2` and `duration = 5`:
```lua
if db.move_mode and not current then
    current, max_charges, start_time, duration = 4, max_slots, GetTime() - 2, 5
end
```
After fix #12, `needs_progress_updates` is:
```lua
local needs_progress_updates = not db.move_mode
    and (current < max_charges and duration and duration > 0 and start_time and start_time > 0)
```
`not db.move_mode` is `false` in move mode, so the whole expression is `false`. The ticker is correctly suppressed. The fake duration of `5` with `start_time = GetTime() - 2` would give `progress Ôëł 0.4` as a static filling-slot visual, which is the intended behavior. No ticker is started; the visual is set once on entry to move mode when `M.refresh()` is called. This is correct.

#### N5. `DRAW_BACKGROUND_IN_FRONT_OF_FRAME` and `DRAW_FILL_IN_FRONT_OF_FRAME` are both `false` but `DRAW_FILL_IN_FRONT_OF_FRAME` is still checked in an `elseif` (sv_bar.lua:43-53, 280-288)

**Severity:** Low / dead code  
**File:** `sv_bar.lua:280-288`

```lua
local DRAW_BACKGROUND_IN_FRONT_OF_FRAME = false
local DRAW_FILL_IN_FRONT_OF_FRAME = false
...
if DRAW_BACKGROUND_IN_FRONT_OF_FRAME then
    ...
elseif DRAW_FILL_IN_FRONT_OF_FRAME then
    ...
end
```

Both constants are `false`, so both branches are unreachable dead code. The `elseif` block and its locals will never execute. This is likely intentional configuration scaffolding for visual tuning, but if the constants are never expected to change, the branch can be simplified to a single default assignment. Low priority.

#### N6. `ensure_decor` uses `SetAtlas(DECOR_ATLAS, true)` (auto-size) but decor frame size is now set from atlas metadata in `apply_layout` (sv_bar.lua:389, 394, 546-547)

**Severity:** Low / minor redundancy  
**File:** `sv_bar.lua:389, 394`

```lua
M.decor_left:SetAtlas(DECOR_ATLAS, true)   -- auto-size
M.decor_right:SetAtlas(DECOR_ATLAS, true)  -- auto-size
```

`SetAtlas(..., true)` auto-sizes the texture to atlas dimensions. Then `apply_layout()` calls `M.decor_left_frame:SetSize(wing_width, wing_height)` on the parent frame, and the texture is `SetAllPoints(frame)` so it stretches to the frame. The auto-size in `SetAtlas` is immediately overridden by the `SetAllPoints` + frame `SetSize` in `apply_layout`. The `true` argument is therefore a no-op in practice. Not a bug; the `false` form would be slightly more precise (`SetAtlas(DECOR_ATLAS, false)` + explicit size), consistent with `set_atlas_sized()` used elsewhere. Very low priority.
