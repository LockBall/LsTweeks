# aura_frames module review — active follow-up, 2026-06-04

This temporary review file tracks only remaining aura_frames review items.

## 1. Higher-Risk Runtime Work

### 1.1 Per-frame aura event ownership
Every runtime aura frame still registers the same events in `af_main.lua`. The shared
scan prevents N full scans. Disabled frames now return from `handle_aura_frame_event()`
before aura-info merging, dirty/cache work, or deferred callbacks, but enabled frames
still use per-frame handlers.

Best options:
- Larger step: centralize aura events on one dispatcher frame and update only enabled
  runtime-active frames.

Risk: CDM refreshes, custom filters, profile loads, and combat deferral all rely on the
current per-frame update params.

### 1.2 `pcall(C_UnitAuras.GetAuraDuration, ...)`
The ticker, render, and scan paths still defensively wrap `GetAuraDuration`.

Remaining call sites:
- `af_core.lua` ticker path
- `af_render.lua` render path
- `af_scan.lua` scan path

Removing the ticker-path `pcall` would be the highest-value performance change, but
only after in-game confirmation that `GetAuraDuration("player", auraInstanceID)` does
not throw on the current Retail client in combat/secret-value scenarios.

### 1.3 CDM identity and cooldown overlay behavior
`hook_cd_item_frame()` and `add_cooldown_viewer_category_entries()` still duplicate
some child identity resolution. `apply_cooldown_overlay()` still tries
`SetCooldownFromDurationObject` with and without the third argument.

Current discrepancy under test:
- Essential frame can show active aura duration then cooldown for Blessing of Freedom.
- Utility frame can show Divine Protection cooldown immediately when cast from already
  in combat instead of active duration first.
- Blessing of Freedom also fails active-first display when moved to Utility.
- User confirmed Essential and Utility are both in cooldown mode in CDM during the
  comparison. Treat this as Utility viewer active-aura ingestion/render precedence
  unless runtime child data proves a settings difference.

Do not refactor this without specific CDM runtime testing. It is coupled to Blizzard
Cooldown Manager child state, hooks, cooldown mode, and secret-value handling.
