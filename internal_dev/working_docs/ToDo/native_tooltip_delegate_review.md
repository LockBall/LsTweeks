Final follow-up plan for commit 6fd4aab (rich Aura tooltips through shared native `GameTooltip` delegates in `functions/tooltip.lua`). The architecture is confirmed correct; preserve it unchanged while hardening tooltip ownership.

Hard boundary: do not reintroduce an addon-created `GameTooltip`, `GameTooltipTemplate`, `securecallfunction`, rendered-line inspection, or `NumLines` checks. Restricted Aura data continues through the shared native delegate; guarded cached/plain lines remain fallback-only.


## Final Plan
1. **Make the tooltip stub ownership-aware.** In `wow_stub.lua`, have `SetOwner(owner, anchor)` retain `self.__owner` as well as recording the call; return it from `GetOwner()`; implement `IsOwned(owner)` as an owner comparison. This is a prerequisite, because the current unconditional `nil`/`false` getters would make an ownership regression pass vacuously. Run the existing tooltip and Aura suites after this stub-only change to expose any assumptions before changing runtime code.
2. **Add a failing ownership-transfer regression.** Exercise the real Aura hover callbacks in `test_af_ranges.lua`: show frame A's native Aura tooltip, transfer the shared `GameTooltip` to frame B with a second `SetOwner`, trigger frame A's `OnLeave`, and assert that no new `Hide` call occurs. Also retain or add the normal case proving that frame A's `OnLeave` hides the tooltip while frame A still owns it. Confirm the transferred-owner case fails against commit 6fd4aab before implementing the fix.
3. **Replace the boolean with explicit ownership.** In `functions/tooltip.lua`, replace `native_tooltip_active` with `native_tooltip_owner`. Set it only after the delegate call succeeds; clear it on delegate failure. Change `addon.HideNativeTooltip(owner)` to act only when `owner == native_tooltip_owner`; call `GameTooltip:Hide()` only when `GameTooltip:GetOwner() == native_tooltip_owner`, then clear the stored owner even if Blizzard or another addon already took ownership. Do not hook global `GameTooltip:OnHide`.
