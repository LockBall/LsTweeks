## Cross-Module Audits
1. [x] Audit `addon.module_defaults` registry writes. Keep entries only when a runtime path actually consumes the shared registry; otherwise let modules own defaults on their local namespace. Player Frame removed an unused `addon.module_defaults.pf` write; a quick grep also showed `addon.module_defaults.ob` written without a reader, while `.st`, `.sv`, and `.audio_volumes` have active consumers. [headless-testable: smoke-load all modules after removing any unused registry write; grep should show at least one consumer for every remaining registry key.] Resolved 2026-07-07: checked all modules. Settings, Skyriding Vigor, and Audio Volumes keep consumed registry keys; Player Frame, Objectives, and Aura Frames own defaults locally. Removed the unused Objectives registry write, and no unused `addon.module_defaults` writes remain.

2. [x] Audit required dependency ownership and fail-visible load-order assumptions. If `.toc` order guarantees defaults, GUI metadata, shared constants, or module runtime tables load first, consumers should read the owned value directly and runtime behavior should call required dependencies directly; keep defensive absence checks only in debug/status paths that intentionally tolerate partial load. Player Frame removed fallback literals for owned constants and now calls `M.fade` directly for runtime behavior while keeping `/lst status` defensive. [headless-testable: smoke-load and exercise enable/disable or event routing after removing each dead fallback or guard.] Resolved 2026-07-07: checked Settings, Player Frame, Objectives, Audio Volumes, Skyriding Vigor, and Aura Frames. Removed dead fallbacks around `.toc`-owned defaults, module metadata, `addon.UPDATE_INTERVALS`, Skyriding Vigor slot/style constants, Audio Volumes target/preset registries, and Aura Frames cooldown/ticker constants. Left optional/status guards, user-key validation, dynamic DB/table initialization, and partial UI-state checks in place.

3. [x] Audit shared control factory footprint constants. Layout callers should not duplicate a factory-owned widget width or height when row/column math depends on that footprint; expose the size from the factory and have callers read it. Player Frame now reads `addon.SLIDER_WITH_BOX_SIZE`; Skyriding Vigor has similar `slider_width` / slider row height layout assumptions around `CreateSliderWithBox` and should be checked when that module is reviewed. [headless-testable: smoke-load settings panels after moving a footprint constant; visual spacing still needs in-game review only if the footprint actually changes.] Resolved 2026-07-07: checked all `CreateSliderWithBox` callers. Player Frame already used `addon.SLIDER_WITH_BOX_SIZE`; Skyriding Vigor, Objectives slider rows, and Audio Volumes situation sliders now read that shared size for column/row math. Aura Frames and Settings do not duplicate the factory footprint in layout math; remaining literal widths belong to non-slider controls or broader panel layout.

4. [x] Audit duplicated setting numeric metadata across modules. Look for min/max/step/default values repeated between defaults files, runtime clamps, GUI controls, and reset/sync helpers; prefer defaults-owned metadata for numeric ranges while GUI files own only labels, control keys, layout, and tooltips. [headless-testable: for modules with public clamp helpers or runtime getters, assert boundary clamping from the centralized metadata.] Resolved 2026-07-07: checked all slider/range call sites. Player Frame and Skyriding Vigor already centralize setting ranges; Objectives keeps small local range tables that feed both clamps and sliders; Settings and Audio Volumes have no duplicated runtime clamp metadata. Aura Frames now owns numeric slider ranges in `M.SETTING_RANGES`, including visible tick, frame position/width, scale/spacing, OOC fade, max icons, short threshold, and timer font size. Added a headless Aura Frames range test for the public visible-icon tick clamp/snap helper.

5. [x] Audit reused epsilon/tolerance constants. Look for constants named for one unit or domain, such as alpha, position, or time, that are also used for another domain; split them so later tuning in one domain does not silently change another. Player Frame split `ALPHA_EPSILON` from `TIME_EPSILON`. [headless-testable: existing boundary tests should still pass after splitting; add targeted tests only if the tolerance boundary itself is meaningful behavior.] Resolved 2026-07-07: checked all modules for named epsilon/tolerance constants and repeated tiny comparison literals. No remaining cross-domain reused tolerance constants found. Named Skyriding Vigor's repeated progress comparison tolerance as `PROGRESS_EPSILON` and its fill-color white comparison as `COLOR_EPSILON`; left domain-specific thresholds such as Aura Frames edge snap and GCD grey threshold unchanged.

6. [x] Audit duration-based animations and timed visual progress for nominal-interval drift. Look for ticker/OnUpdate code that advances progress by a fixed interval per callback; when a user-facing duration should mean real elapsed time, prefer cached `GetTime()` start timestamps. Do not apply this to debounces, event batching, polling, or retry timers where exact visual duration is not the contract. [headless-testable: where the stub clock can skip ahead, assert a delayed tick catches up to the configured duration instead of requiring every nominal interval callback.] Resolved 2026-07-07: checked Player Frame, Skyriding Vigor, Aura Frames, Objectives, Audio Volumes, shared slider/debounce helpers, and core delayed refresh paths. User-facing duration code already uses elapsed-time or `GetTime()`-based progress, aura expiration timestamps, or WoW duration objects. Remaining `C_Timer.After`, `NewTimer`, `NewTicker`, and `OnUpdate` paths are debounces, preview restore delays, event buckets, hover checks, retry/follow-up refreshes, drag polling, or status maintenance where nominal cadence is the intended behavior.

7. [x] Audit debounce and delayed-work preconditions. Look for event handlers that allocate/cancel debounce timers even when the eventual callback will immediately no-op because of state, enablement, threshold, visibility, combat, or missing-target gates; move cheap precondition checks before timer allocation where behavior stays equivalent. [headless-testable: fire events in no-op states and assert no debounce timer remains queued, then fire in the one actionable state and assert the timer is queued.] Resolved 2026-07-07: checked Player Frame health debounce, Objectives title/background/auto-collapse queues, Aura Frames deferred scans/cooldown refresh/tooltip retry/hover checks, Audio Volumes preview timers, and the shared slider debounce. Player Frame, Aura Frames, Audio Volumes, and shared slider paths already gate cheap no-op states before queuing or intentionally queue only while active. Objectives auto-collapse now checks `should_auto_collapse()` before allocating the next-frame collapse timer, protecting settings/internal callers from queuing work that the callback would immediately skip.

8. Audit no-op refresh, teardown, and restore paths. Look for refresh/update/stop handlers that restart tickers, OnUpdate loops, scans, animations, frame lookups, or restore calls when current state already equals the target; skip or collapse the work when the handler can apply the final state once or prove the feature was inactive. [headless-testable: drive the module to the steady state, trigger its public refresh/stop path, and assert no new ticker/OnUpdate/scan work remains queued and no unnecessary restore path is called.]

9. Audit state-machine helper functions for hidden caller-order dependencies like Player Frame latent trap 1. Look for functions that set runtime state to idle/normal while combat, enablement, visibility, or lifecycle flags still indicate a guarded state, and verify they are correct without relying on the caller to immediately run a full refresh. [headless-testable: the lua_tests harness can call each state helper in isolation (no follow-up refresh) and assert the resulting state against the combat/enablement flags — pin each audited helper with a test as it is reviewed.]
